# 07. 출시 통제·자문 의존성 (Release Control & Advisory Dependencies)

> 본 문서는 베타 출시 전 충족해야 할 출시 게이트와 그 게이트의 외부 자문 의존성을 단일 출처로 정리한다.
> Phase 완료가 베타 출시 가능 상태를 의미하지 않는다.
> 본 문서는 정책 00·01·04·05·06과 상위 충돌하지 않는다(00이 항상 우위).

---

## 1. 출시 통제 원칙

> **"개발 속도보다 출시 통제력이 경쟁력이다."**

Phase 7~10이 코드상 완료되었어도, 본 문서의 게이트가 미충족 상태이면 베타를 시작하지 않는다.

세 가지 부정 원칙(절대 하지 않음):

1. 자문 미수령 상태에서 그 자문이 차단하는 Phase를 시작하지 않는다.
2. 미충족 게이트가 있을 때 우회 코드를 추가하지 않는다.
3. 「데모데이 일정」을 이유로 게이트를 약화시키지 않는다.

## 3. Phase별 진행 가능 조건 (Phase Gate)

본 절은 Phase 7~10의 시작·종료 조건을 정의한다.

### Phase 7 (룰 엔진)

- **시작 차단**: G6 자문(C1·C4) 미수령. C1은 실제 KEPCO YAML 단가 확정, C4는 100건 케이스 설계.
- **종료 조건**: tariff_engine.py 구현 + 100건 합성 테스트 통과 + 데모 YAML이 「DEMO_ONLY」 경고 유지.
- **현재 단계**: 자문 대기. Phase 7a(자문) → Phase 7b(구현)로 분할.

### Phase 8 (전체 가드레일 통합 + 시각 마스킹)

- **시작 조건**: Phase 7 자문 진행 중 병행 가능(자문 대기 시간을 활용).
- **종료 조건**: 시각 마스킹(redaction) 구현 + 모든 사용자 출력 경로에 `assert_safe_output` 적용 + 부정 문맥 가드레일 정교화.
- **시급도**: 가장 위험한 내부 결함(D8)을 잡는 단계.

### Phase 9 (자료요청 메시지 예시 + 자료 요약본 PDF)

- **시작 차단**: G5 자문(A1·A2·A3·A5) 미수령. 시각 마스킹(Phase 8) 미완료.
- **종료 조건**: 메시지 예시는 자동 발송 기능 없이 「복사」만 제공 + 자료 요약본 PDF에 시각 마스킹 적용 + 정책 01의 신규 메시지 표현 범위 통과.
- **명칭**: 「증거 패키지」가 아닌 **「에너지 청구 자료 요약본」**.

### Phase 10 (KPI metrics + 데모 데이터)

- **시작 조건**: 다른 Phase와 병행 가능.
- **종료 조건**: 새 KPI 계층(정책 06 §3)이 events 테이블에서 계산 가능 + scripts/export_metrics.py 동작.
- **NGO 자문(E1~E5)**: 본 Phase 전에 받아 운영 매뉴얼에 반영.

## 4. 자문 질문 리스트 (자문서에 그대로 들고 갈 것)

### A. 변호사법 자문 (화우 ESG센터)

A1. 빈칸형 자료요청 메시지 예시는 변호사법 §109의 「법률관계 문서 작성」에 해당하는가? 로폼 판결의 「사실 입력의 기계적 반영」 영역에 속하는가?

A2. 한전 전기소비자보호센터·주택임대차분쟁조정위 신청 절차 안내문에서 어디까지가 「개별 사안 적용」인가? 정적 markdown 안내가 안전 영역인가?

A3. 검산 결과 화면의 「공식 산식 기준 계산값과 사용자 청구액 사이에 차이가 확인됩니다」 표현은 명예훼손·업무방해 리스크에서 안전한가? 차이액을 숫자로 표시하는 것 자체는 사실 진술인가?

A4. 분쟁 단계에서 AI가 사건 검토 후 변호사에게 자료 패키지를 넘기는 구조는 변호사법 §109 위반인가? AI 사건 검토 없이 사용자를 변호사에게 직접 연결하는 구조는 안전 영역인가?

A5. 본 출시 게이트 G5의 사전 자문 의견서가 어떤 형식·범위·서명자를 갖춰야 PoC 출시 정당성을 확보할 수 있는가?

A6. 메시지 예시에 허용되는 표현: 「정산 재확인」·「자료 확인 요청」·「산식 공개 요청」. 금지되는 표현: 「과다」·「반환」·「법적 조치」. 이 분류가 적절한가?

### B. 한전 요금 자문 (전기요금 전문가, 전 한전 출신)

B1. 주택용 저압 누진제 3단계 단가, 기본요금, 기후환경요금, 연료비조정단가, 부가세, 전력산업기반기금. 2026년 5월 시점 적용 약관의 단가는?

B2. 한전 검침일(월말 기준)과 임대인 청구 기간이 다를 때 환산 방식은? 일할 계산이 표준인가? 비례 배분 방식의 정확한 산식은?

B3. 다가구·고시원에서 임대인이 사용하는 세대별 배분 산식의 표준 유형은? 사용량 기반·면적 기반·인원 기반·평등 분배 중 어느 것이 가장 흔하며, 각각의 산식 표기는?

B4. 합성 테스트 100건의 케이스 설계: 누진제 구간 경계값(199/200/201, 399/400/401 kWh), 검침일 모서리(월말 ±3일), 계약종별 다양화, 부가세·기금 반올림 규칙. 어떤 100건이 실제 약관을 충분히 커버하는가?

B5. 한전 약관 변경 주기와 자동 모니터링 가능성. tariff YAML 버전 관리 + 신뢰가능한 source URL은?

### C. 개인정보 자문 (외부 검토)

C1. 외부 LLM API(Anthropic 미국·OpenAI 미국)로 임차인+임대인 개인정보가 포함된 이미지가 송신됨. 개보법 §28 국외 이전 별도 동의가 필요한가? 동의 받지 못한 사용자에 대해 수동 입력 모드만 제공하는 것이 합법적 우회인가?

C2. 임차인이 업로드한 임대인 정보(이름·계좌·연락처)를 임대인 동의 없이 처리하는 법령상 근거는? 개보법 §15 ②, §17 중 어느 것?

C3. 임대차계약서에 주민번호 포함 시: OCR 단계에서 자동 검출하여 **업로드 자체를 거부**하는 정책이 안전한가? 마스킹 후 보관 vs 업로드 거부 중 권장 사항은?

C4. 안전체크 응답(폭언·협박·보복 우려)은 §23 민감정보에 해당하는가? 별도 동의 필요한가?

C5. ISMS-P 인증 필요 시점은? 베타 100명 규모에서 정보통신서비스 제공자 분류 가능성?

### D. NGO 자문 (주거 NGO)

D1. 안전체크 6문항의 실제 변별력은? 누락 문항·삭제 문항 권고?

D2. HIGH 등급에서도 사용자가 「메시지 예시 보기」를 강행할 때의 대응 정책. 현재 정책: 메시지 복사 차단 + 안내 표시. 권고: VERY HIGH 등급에서는 메시지 자체를 숨김 처리.

D3. 외국인 노동자 한국어 접근성. 다국어 후속 단계의 우선순위 언어(베트남·네팔·우즈베크어 중)?

## 6. 자문 결과 반영 절차

자문 결과가 들어오면:

1. 의견서를 `docs/advisory/{date}_{topic}.md`에 보관.
2. 영향받는 정책 문서(00~06)를 보강하고 변경 이유에 자문 일자·자문 주체를 명시.
3. 코드 영향이 있으면 해당 Phase의 작업 항목에 추가.
4. 회귀 테스트를 보강하여 자문 결정이 코드 수정으로 깨지지 않도록 차단.

| 「산식 차이」 표현 | "공식 산식 기준 계산값과 사용자 청구액 사이에 차이가 확인됩니다" + 면책 부착 | A3 자문서로 표현 강도 조정 |
| 변호사 연결 구조 | AI 사건검토 없이 사용자가 직접 자료 요약본을 가져가는 구조 | A4 자문서로 화우 pro bono 채널 워크플로 확정 |
| G5 의견서 형식 | 조건부 검토 메모 (포괄 의견서 아님) | A5 의견서 수령 시 게이트 충족 |

### 8.2 한전 전기요금 자문 (C1·C2·C3·C4·C5) — Phase 7b 차단 유지

- **C1**: KEPCO YAML 실제 단가 미확정. tariff_engine 구현 보류.
- **C2**: 검침기간 일수 보정 — "일할 계산은 마지막 수단" 원칙만 정책 03에 반영. 코드 구현은 자문 후.
- **C3**: 세대별 배분 산식 7유형(개별 보조계량기형 / 동일 분배형 / 면적 비례형 / 인원 비례형 / 정액 부과형 / 공용부 포함형 / 임의 청구형) — 분류 기준만 정책 02에 반영 가능, 검산 로직은 자문 후.
- **C4**: 합성 100건 케이스 분포안 (구간 25·계절 15·검침기간 15·부가항목 15·계약종별 10·배분 15·오류 10) — 자문 후 확정.
- **C5**: tariff YAML 메타데이터 표준 (source_url, retrieved_at, valid_from/to, reviewer, checksum) — 정책 03 §2에 반영.
